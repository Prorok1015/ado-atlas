# Спецификация: архитектура и модульность

Покрывает: C02, C03, C16, C30, C37, C39, C40, C22.

## Позиция по «переписать на React/TypeScript»

Не надо. Отсутствие бандлера и фреймворка — **осознанный трейд-офф**, зафиксированный
в `ARCHITECTURE.md`, и он окупается: нулевой build-step, мгновенная отладка в Chrome,
никакого supply-chain через npm-дерево. Для соло-разработчика это рационально.

Но у трейд-оффа есть **цена**, и её надо платить осознанно: класс ошибок, который бандлер
и типы ловят бесплатно (порядок загрузки, TDZ, опечатка в имени, битая ссылка на глобал),
здесь **не ловится ничем** до рантайма. `ARCHITECTURE.md` это честно признаёт в разделе
«Hard lessons». Задача — не отменить трейд-офф, а **вернуть себе гейт**, не заводя бандлер.

## A1. Гейт на полноту и порядок скриптов · high · 1 день

**Где:** `index.html:697-766`, `tools/check-globals.js:27-31`, `src/app.js:467-477`

~40 `<script>` в строго заданном порядке. `check-globals.js` сканирует каталог рекурсивно
и просто компилирует конкатенацию — он ловит **дубли имён и синтаксис**, но:

- **не проверяет, что файл вообще подключён** в `index.html` → новый файл, забытый в разметке,
  проходит `npm test` зелёным и падает только в браузере;
- **не проверяет порядок** → перестановка тега даёт TDZ;
- **не ловит парс-тайм зависимости**. А они есть:

```js
// app.js:467-477 — на ВЕРХНЕМ уровне, во время парсинга
const parentEditor   = createParentField('s_parent', ...);   // ← зависит от card-picker.js:359
      assignedEditor = createAssigneeField(...);             // ← и от bare-глобала sprintRoot
```

`ARCHITECTURE.md` уже фиксирует «Parse-time gotcha» — то есть на эту мину **уже наступали**
и обезвредили вручную. Гейта, который поймал бы следующую, до сих пор нет.

**Сделать — `tools/check-scripts.js`, в `npm test`:**

1. **Полнота.** Каждый `src/**/*.js` должен быть либо в `index.html`, либо в
   `background.js importScripts`. Иначе — ошибка с именем файла. Это ловит самый частый
   и самый глупый способ сломать сборку.
2. **Порядок.** Ввести лёгкую декларацию зависимостей комментарием в шапке файла:
   ```js
   // requires: namespace, state-globals, card-picker
   ```
   Скрипт топологически сортирует и сверяет с фактическим порядком в `index.html`.
   Расхождение — ошибка. Дёшево, без бандлера, самодокументируемо.
3. **Заодно** — генерировать раздел «Load order» в `ARCHITECTURE.md` из `index.html`,
   чтобы документация не дрейфовала (см. A6).

**Плюс:** вынести создание пикеров из парс-тайма в явную `initPickers()`, вызываемую
из `initialBoot`. Парс-тайм side effects — источник именно тех TDZ, от которых нет защиты.

**Критерий приёмки:** новый файл, не добавленный в `index.html`, роняет `npm test`.
Перестановка `<script>` в неверный порядок роняет `npm test`.

## A2. Разрезать модули-гиганты · high · поэтапно, 2-3 недели

**Где:**

| Файл | Строк | Что внутри намешано |
|---|---|---|
| `filter-builder-modal.js` | 2628 | DOM + CRUD сохранённых фильтров + вся проводка событий. `ensureModalElement()` — **~900 строк** (1165–2082). 38 присваиваний `innerHTML`, 63 инлайновых `.onclick` |
| `layout.js` | 1725 | persist тулбара/сайдбара + движок раскладки + drag-drop конструктор (`getMockFieldHtml` 911-1073) |
| `ai-search-dialog.js` | 1556 | модалка + фоновая карточка + полный CRUD настроек `AISettingsDialog` |
| `ai-search-service.js` | 1389 | загрузка схемы + эвристика уровня + 3 пайплайна + ~200-строчный JSON-парсер (:531) + нормализация IR |
| `side-panel.js` | 1348 | |

**Почему это high, а не «ну большие файлы».** Для соло-разработчика это **главный источник
регрессий**: любая правка рискует задеть соседнюю подсистему в том же файле, юнит-тест
написать не на что (всё сцеплено с DOM), а ревью — только глазами по 900-строчной функции.
Это ровно тот техдолг, который тормозит все остальные пункты роадмапа.

**Сделать — резать по швам, без фреймворка, по одному файлу за раз:**

- `filter-builder-modal.js` → выделить **`saved-filters.js`** (строки 1211-2082 — это
  самостоятельный CRUD с собственным состоянием). Самый чистый шов, начать с него.
- `layout.js` → **`layout-preview.js`** + **`layout-builder.js`** (конструктор отделён
  от движка раскладки).
- `ai-search-dialog.js` → **`ai-settings-dialog.js`** (CRUD настроек — независимая сущность).
- `ai-search-service.js` → **`schema-builder.js`**, **`json-utils.js`**, **`ir-normalizer.js`**.
  Это чистые функции — **и их сразу можно покрыть тестами**, что и есть главный выигрыш.

**Правило разреза** (из `ARCHITECTURE.md`, оно верное): если символ вызывается **bare**
из нетронутого модуля — он остаётся bare, файл просто переезжает. Не намespace'ить ради
красоты.

**Порядок:** по одному файлу в спринт, каждый — отдельный коммит с прогоном `npm test`
и ручным smoke. Не пытаться сделать всё сразу.

## A3. AI-запросы без таймаута и отмены · high · 1 день

**Где:** `custom-cloud-provider.js:155`, `background.js:187`, `ai-search-dialog.js:219,1473`

```js
// custom-cloud-provider.js:155 — signal ЕСТЬ, но в воркер не передаётся
chrome.runtime.sendMessage({ action: 'fetchCloudAI', url, method, headers, body });
//                                    ↑ options.signal ПОТЕРЯН, хотя _callAPI его выставляет (:214,:258)

// background.js:187 — ни AbortController, ни таймаута
fetch(url, { method, headers, body });
```

А в `ai-search-dialog.js` **нет ни одного `AbortController`**: `cancelBtn` и закрытие окна
(`:219`, `:1473`) просто **прячут окно**, запрос продолжает жить.

**Последствия:** зависший облачный endpoint держит запрос бесконечно, канал `sendMessage`
не закрывается, UI застревает на «Generating filters», отмены нет. Для BYOK-пользователя
это ещё и **деньги**: он платит за запрос, который уже никому не нужен.

**Сделать:**
1. `AbortController` в `handleSubmit`, `signal` пробросить до реального `fetch`.
2. Через `sendMessage` сигнал не передать — использовать **`chrome.runtime.connect`** (порт):
   разрыв порта = сигнал отмены в воркере. Либо слать отдельное сообщение `abort` с id запроса.
3. В `background.js` обернуть `fetch` в `AbortSignal.timeout(N)` (разумно — 60 с) и слушать отмену.

**Критерий приёмки:** Cancel действительно прерывает сетевой запрос; зависший endpoint
отваливается по таймауту с внятной ошибкой.

## A4. Сохранённые фильтры мимо `App.prefs` · medium · 4 часа

**Где:** `filter-builder-modal.js:1238,1256-1263,215,2363,2596`

```js
const storage = (window.chrome && chrome.storage && chrome.storage.sync) || null;
storage.set({ fbSavedFilters });                       // ← напрямую, мимо App.prefs
// fallback: localStorage.setItem('fbSavedFilters', ...)
```

`ARCHITECTURE.md:31` объявляет `App.prefs` «the single canonical store for persisted prefs».
Ключи `fbSavedFilters` / `fbDraftFilter` это правило **нарушают** — и теряют всё, что
`App.prefs` даёт бесплатно: per-key LWW по timestamp, роуминг между устройствами,
попадание в `export()`/`import()`, покрытие тестом `check-prefs.js`. Плюс dual-write
переизобретён инлайном.

Обидно, потому что `App.prefs` — **лучшая подсистема проекта**, и её просто не использовали.

**Сделать:** перевести `fbSavedFilters` / `fbDraftFilter` на `App.prefs` (статический
sync-ключ либо dynamic-key facility — механизм уже есть, так сделан `ado.layout.<wtype>`).
Самописный storage-fallback удалить.

**Критерий приёмки:** сохранённые фильтры роумятся между устройствами и попадают в `export()`;
`check-prefs.js` их видит.

## A5. Три копии `extractJSON` и бессмысленный fallback · medium · 3 часа · (UNVERIFIED)

**Где:** `ai-search-service.js:531`, `chrome-prompt-provider.js:208`, `custom-cloud-provider.js:117`

`extractJSON` реализован **трижды**: робастная версия (~200 строк: балансировка скобок,
срез комментариев, trailing commas) — в сервисе; упрощённые — в обоих провайдерах. Итог:
облачные провайдеры получают **заметно более слабый** разбор JSON, чем on-device путь.

Плюс явный баг:
```js
// custom-cloud-provider.js:104
catch (e) { console.warn(...); return JSON.parse(this.extractJSON(raw)); }
//                                    ↑ ТОТ ЖЕ вызов, который только что бросил → бросит снова
```
Это не fallback, это копия падения.

**Сделать:** один общий `AdoLib.extractJSON` (взять робастную версию), переиспользовать
во всех трёх местах — **и покрыть тестами**, он чистый. В `custom-cloud` сделать настоящий
repair-проход (как у Chrome-провайдера) либо пробросить осмысленную ошибку.

## A6. Синхронизировать `ARCHITECTURE.md` с кодом · low · 2 часа

Документ хороший и живой, поэтому расхождения особенно вредны — им верят.

- **`ARCHITECTURE.md:96`** описывает порядок загрузки `namespace → const → loading → badges
  → state-globals → undo …`, но реальный `index.html:732-739` вставляет между ними
  `analytics`, `prefs` и `backend` — в доке их нет.
- **`ARCHITECTURE.md:112`** утверждает: «Every new `src/**/*.js` must be added to
  `tools/check-globals.js` AND `index.html`». Первая половина **неверна** —
  `check-globals.js:27` сканирует каталог рекурсивно, вручную добавлять ничего не нужно.
  Разработчик по этой инструкции делает лишнюю работу.
- **§10** подаёт RTL как реализованную фичу — [это не так](07-spec-accessibility-i18n.md).
- **§9** описывает Analytics Dashboard и Ultra-Dark как существующие — [их нет](08-spec-monetization.md).

**Сделать:** синхронизировать; идеально — **генерировать** раздел Load order скриптом из
`index.html` (тогда он не сможет разойтись). Статусы фич в §4/§9 привести к факту.

## A7. Мульти-вендорность: честно назвать заглушкой · low · 1 час

`App.backend` (`backend.js`) — реестр провайдеров с дескрипторами `capabilities` /
`connectionSchema` / `fieldSchema`, а `FilterIR` + `filter-compiler.js` действительно
изолируют компиляцию в WIQL. Заготовка **спроектирована правильно**.

Но у неё **нет ни одного потребителя**: `api.capabilities` в UI не запрашивается, второго
провайдера нет. `ARCHITECTURE.md §3` («Capability-Driven UI») описывает это как работающий
механизм — а это **статичная заглушка**.

Это не дефект: дисциплина `FilterIR` полезна сама по себе и её стоит сохранить.
Дефект — в документации, которая выдаёт намерение за факт.

**Сделать:** в `ARCHITECTURE.md` явно пометить §3 как «design intent, not yet wired».
Второго провайдера (Jira/GitHub) [роадмап сознательно откладывает](10-roadmap.md) за
горизонт 12 месяцев — второй бэкенд удваивает поверхность поддержки для одного человека.
