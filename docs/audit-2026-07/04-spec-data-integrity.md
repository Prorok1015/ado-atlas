# Спецификация: целостность данных

Покрывает: C04, C05, C06, C20, C22, C23, C41.

Это самый недооценённый раздел аудита. Ошибки здесь **тихие**: пользователь не видит
ни сообщения, ни исключения — он просто через неделю обнаруживает, что правка коллеги
исчезла. Такие баги разрушают доверие к инструменту сильнее, чем падения.

---

## D1. Записи без проверки ревизии — тихий last-write-wins · high · 1 день

**Где:** `items.js:325-343` (`updateItem`), `bulk.js:403-419`, `time.js:106-111`

```js
// updateItem — собирает ops БЕЗ теста ревизии
ops.push({ op: 'add', path, value: v });
// → PATCH .../workitems/${wid}
```

Самое обидное здесь то, что **проект уже умеет делать правильно**. Рядом, в том же файле:

```js
// setParent (items.js:504), addDependency — делают как надо:
{ op: 'test', path: '/rev', value: d.rev }
```

И редактор **уже хранит ревизию**: `editor.js:444` — `nodes[id].rev = r.rev`. Она просто
не доезжает до запроса.

**Что происходит на практике.** Ты открыл задачу в ADO Atlas, коллега в это же время
правит её в нативном портале (или ты сам — в соседней вкладке). Ты жмёшь Save. Твой PATCH
без `test /rev` **молча затирает** его изменение. Ни конфликта, ни предупреждения —
изменение просто исчезает. То же самое в bulk: `bulk.js:428-441` сводит результат к
счётчикам `ok++ / fail++`, **без списка проблемных id и причин** — пользователь видит
«12 ok, 3 fail» и не знает, что именно не прошло.

**Сделать:**

1. Первой операцией в массиве — тест ревизии:
```js
const ops = [{ op: 'test', path: '/rev', value: rev }, ...fieldOps];
```
   `rev` брать из `nodes[id].rev`, который уже есть.

2. На `HTTP 409` — не глотать, а показать баннер конфликта:
   «Элемент изменён другим пользователем. [Перезагрузить] [Перезаписать]».
   «Перезаписать» = повторить PATCH с новым `rev` (осознанное решение пользователя).

3. В bulk — агрегировать per-item ошибки. `$batch` возвращает `code` и `body` **на каждый
   элемент**, эта информация сейчас просто выбрасывается. Собрать в отчёт: список id
   + причина, с кнопкой «повторить неудавшиеся».

**Критерий приёмки:** правка элемента, изменённого параллельно, приводит к 409 и диалогу
конфликта, а не к тихой перезаписи. Bulk показывает, какие именно id не прошли и почему.

## D2. Ретрай неидемпотентных POST → дубликаты · high · 4 часа

**Где:** `http-auth.js:125-155`

Единый цикл ретрая применяется ко **всем** HTTP-методам:
```js
if (err.name === 'AbortError') throw err;
if (attempt < MAX_RETRIES) { await sleep(...); continue; }   // ← и для POST тоже
```

Через этот `req()` идут: `createItem` (`items.js:485`), `comment` (`items.js:354`),
`createSprint` (`endpoints.js:87`) — всё это **POST**.

**Сценарий потери.** Сервер принял `POST /workitems` и создал задачу, но ответ не дошёл
(обрыв сети, 502 от балансировщика уже **после** записи). Клиент видит ошибку, ретраит —
и создаёт **вторую** задачу. Пользователь получает дубликат и не понимает, откуда он.
То же с комментариями (два одинаковых комментария) и спринтами (два iteration-node).

Ирония в том, что ретрай — одна из сильных сторон проекта (честный `Retry-After`,
экспоненциальный бэкофф с джиттером). Просто он применён шире, чем безопасно.

**Сделать:**
```js
const IDEMPOTENT = new Set(['GET', 'PUT', 'DELETE', 'HEAD']);
const canRetry = IDEMPOTENT.has(method) || (method === 'PATCH' && hasRevTest(body));
```
- `GET`/`PUT`/`DELETE` — ретраить как сейчас.
- `PATCH` **с** `test /rev` (после D1) — ретраить безопасно: повтор либо пройдёт,
  либо честно упадёт с 409.
- `POST` — **не ретраить**. Показать пользователю «не удалось создать, повторить?» —
  пусть решает он, зная контекст.

Различать «ошибка до отправки» (сеть не поднялась — ретрай безопасен) и «ответ потерян»
(запрос ушёл — ретрай опасен) на уровне fetch нельзя, поэтому для POST консервативная
стратегия — единственная корректная.

**Критерий приёмки:** POST не ретраится автоматически; создание задачи при обрыве сети
не приводит к дубликату.

## D3. Глобал `detectedTargetField` флиппается между типами · high · 4 часа

**Где:** `http-auth.js:173-177`, `core.js:157-159`, `items.js:223-238`

```js
// mapWorkItem пишет в ОБЩИЙ модульный глобал — по последнему обработанному item
if (FIELD_REGISTRY.finish.ref in f)      detectedTargetField = FIELD_REGISTRY.finish.ref;
else if (FIELD_REGISTRY.target.ref in f) detectedTargetField = FIELD_REGISTRY.target.ref;
```

В проекте со смешанными типами (у части элементов `FinishDate`, у части `TargetDate` —
обычная ситуация для кастомных процессов) значение **зависит от порядка обработки**
в `batchFetch`. А порядок недетерминирован в том смысле, что batch возвращает поле,
только если у элемента есть значение.

**Последствие:** Timeline/Gantt может читать не то поле, а сохранение целевой даты —
**писать не в то поле** у части элементов. Данные уходят не туда, и это выглядит как
«Gantt глючит», а не как баг записи.

**Сделать:** убрать глобал. Разрешать target-поле **по конкретному `wtype`** — механизм
уже есть, `getWorkItemTypeFields` кэширует набор полей типа. Передавать resolved-ref
явно в `mapWorkItem` / `updateItem`, а не через модульную переменную.

**Критерий приёмки:** в проекте, где Bug использует `TargetDate`, а Feature — `FinishDate`,
обе даты читаются и пишутся корректно независимо от порядка загрузки.

## D4. Молчаливое проглатывание ошибок · medium · 2 дня

**Где:** повсеместно. **77 пустых `catch`** (61 именованный).

```js
endpoints.js:18-21   iterations  catch(_) { return []; }
endpoints.js:40-42   areas       catch(_) { return []; }
endpoints.js:5-14    me()        catch(_) { return ''; }
items.js:425         comments    catch(_) { ... }
```

**Проблема не в самих catch, а в том, что они делают ошибку неотличимой от пустоты.**
HTTP 403 (нет прав), 404, обрыв сети, троттлинг — всё превращается в `[]`. Пользователь
видит доску без спринтов и думает, что спринтов нет. На самом деле у него не хватает прав,
и он никогда об этом не узнает.

Особо опасный случай — **undo/redo**:
```js
bulk.js:148   try { await api.setParent(o.id, o.old); } catch(e) {}
              afterUndo(null);   // ← вызывается БЕЗУСЛОВНО
```
API-вызов упал, а UI показывает «откат применён». Локальное состояние разошлось с сервером —
и пользователь работает с ложной картиной мира. Это хуже, чем явная ошибка.

Отдельно: `init.js:116` — **один** `try` обхватывает **всю** гидрацию настроек до `:184`.
Одна битая настройка молча роняет весь остаток инициализации.

**Сделать:**

1. Различать «нет данных» и «сбой». Минимально инвазивно — возвращать `{ data, error }`
   вместо голого `[]`, и показывать в UI плашку «не удалось загрузить спринты (403)»
   вместо пустого списка.
2. Логировать проглоченное в `console.warn` — хотя бы для диагностики по репортам.
3. **Undo/redo:** хелпер, который считает неуспехи и при любом из них **не** рапортует
   об успехе, а триггерит `refresh()`, чтобы вернуть UI в согласованное состояние.
4. Пустые `catch` оставить **только** для честно best-effort операций: телеметрия,
   `revokeObjectURL`, аналитика. Там молчание — правильное поведение.
5. Разбить `init.js:116` на автономные `try` по блокам.

**Критерий приёмки:** 403 на загрузке спринтов показывает пользователю причину;
неудавшийся undo не рапортует об успехе.

## D5. Кап 2000 и тихая потеря элементов · medium · 1 день

**Где:** `core.js:10` (`LIST_CAP = 2000`), `query.js:28,69-76`

Две отдельные проблемы, живущие в одном месте:

**Кап.** Крупные проекты видят максимум 2000 элементов. Единственный сигнал —
булев флаг `truncated`. Дерево, граф и Gantt строятся по **неполному набору**, и связи
с необрезанной частью не видны.

**Тихая потеря.** `batchFetch` шлёт `errorPolicy: 'omit'`, а затем:
```js
ids.map(i => byId[i]).filter(Boolean)   // ← молча выкидывает не вернувшиеся id
```
Любой id, по которому `workitemsbatch` вернул ошибку (нет прав, элемент удалён), просто
**исчезает**. При этом `truncated` считается только по достижению 2000 — то есть
количество может не сойтись, и **это никак не сигнализируется**.

**Сделать:**
- Считать и показывать число `omit`-элементов **отдельно** от `truncated`: «показано 1847
  из 1900; 53 недоступны (нет прав)». Это отличает обрезку от проблемы доступа — сейчас
  их не различить.
- Для капа: либо курсорная дозагрузка поверх 2000 (WIQL по батчам id), либо честный
  постоянный баннер «>2000, показаны первые N — сузьте фильтры».

**Замечание по продукту.** В [роадмапе](10-roadmap.md) снятие капа сознательно **отложено**:
масштаб рассматривается как Pro-сигнал, а боль крупных команд адресуется B2B-моделью.
Но **честное сообщение** о том, что данные неполны, — не фича, а корректность. Делать сейчас.

## D6. Кэши без TTL и риск переполнения квоты · medium · 1 день

**Где:** `endpoints.js:146-176`, `:370-431`, `core.js:141-146`, `snapshot.js:17-23`

**Схема без TTL.** `global_fields_map_v4:*` и `wit_fields_v7:*` кэшируются в
`chrome.storage.local` **бессрочно**. Новое поле, новое состояние или новые
`allowedValues` в процессе ADO **не появятся** у пользователя, пока он вручную не почистит
storage или разработчик не поднимет версию ключа. Пользователь будет думать, что расширение
сломано.

Хуже: `setConfig` (`core.js:141-146`) при смене проекта обнуляет **только память**:
```js
teamRosterCache = null; globalFieldsCache = null; populatePromise = null;
// storage-ключи НЕ трогаются
```
→ переключился на другой проект — получил схему полей от предыдущего.

**Квота.** `snapshot.js:17-23` при `roots > 1500 || nodes > 4000` пишет **весь**
`App.state.store.nodes` — **вместе с сырыми `mapped.fields`** (`http-auth.js:227`) —
в `chrome.storage.local`. До 4000 узлов с полными raw-fields упираются в квоту (~10 МБ),
`set()` падает, **ошибка молча глотается** — и «мгновенный первый рендер», главная
фича восприятия скорости, тихо перестаёт работать. Никто не заметит.

**Сделать:**
1. TTL + ревалидация схемы (сутки), плюс кнопка «обновить метаданные проекта» в настройках.
2. `setConfig` должен инвалидировать **и storage-ключи** схемы, не только память.
3. В снапшот сохранять **только лёгкие поля** (`id`, `type`, `title`, `state`, `parent`,
   `rev`), а не сырой `fields`. Это радикально уменьшает объём.
4. Ловить переполнение квоты явно, логировать и деградировать **заметно** (снапшот
   отключён — так и сказать), а не молча.

**Критерий приёмки:** новое состояние в процессе ADO появляется в фильтрах в течение суток;
смена проекта не подтягивает чужую схему; снапшот на 4000 узлах не падает по квоте.

## D7. Литерал `'me'` уходит в ADO · low · 1 час · (UNVERIFIED)

**Где:** `endpoints.js:5-14`, `items.js:260-263,464`

```js
me()  → catch(_) { return ''; }            // при любой ошибке — пустая строка
// items.js:
if (fields.assigned === 'me') { const u = await me(); if (u) fields.assigned = u; }
//                                          ↑ если u === '', остаётся литерал 'me'
fields.assigned = (assigned === 'me' ? (await me()) || assigned : assigned);  // '' || 'me' = 'me'
```

Если `connectionData` не ответил (сеть, узкий scope PAT), «Assign to me» отправляет в ADO
строку **`'me'`** как identity. ADO вернёт невнятную ошибку резолва — пользователь увидит
непонятный сбой сохранения вместо назначения на себя.

**Сделать:** кэшировать текущего пользователя (`currentUser` уже есть в app-state) и падать
с понятным сообщением «не удалось определить вас — повторите», а не отправлять литерал.
