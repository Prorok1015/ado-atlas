## Часть I. Доступность (a11y)

### Масштаб проблемы

Цифры получены grep'ом по всему `src/` + `index.html`:

| Что | Сколько |
|---|---|
| Атрибутов `aria-*` | **2** (оба — `aria-label="Close"`) |
| Атрибутов `role=` | **0** |
| Реализаций focus-trap | **0** |
| Элементов `<h1>` на одной странице | **8** |

**Продукт в Chrome Web Store фактически недоступен для незрячих.** Это не «недоработка
полировки» — это класс пользователей, который не может им пользоваться вообще.

Хорошая новость: это чинится **централизованно**. Почти вся модальная логика уже проходит
через `LayerManager` — значит, одна правка в нём покрывает все ~10 оверлеев сразу.

### X1. Семантика диалогов и сегмент-контролов · high · 1 день

**Где:** `index.html:657` и все `*-overlay`, `index.html:33`

```html
<!-- сейчас: скринридер не знает, что это диалог -->
<div id="confirm-overlay" class="modal-backdrop">
<!-- то же у setup-overlay, newitem-overlay, customize-overlay, sprint-overlay… -->
```

**Сделать:**
```html
<div id="confirm-overlay" class="modal-backdrop"
     role="dialog" aria-modal="true" aria-labelledby="confirm-title">
```
Заголовки в модалках **уже есть** — нужно только проставить им `id` и сослаться через
`aria-labelledby`. Работы на час.

**Сегмент-контролы** (Tree / Graph / Board / Timeline, `index.html:33`):
```html
<div class="seg" id="mode">          <!-- сейчас: role нет, выбор не объявляется -->
```
→ `role="tablist"` на контейнер, `role="tab"` + `aria-selected` на кнопки. Синхронизировать
`aria-selected` с классом `.on` **одним хелпером** — сейчас класс переключается в нескольких
местах (`init.js` wireControls, `settings.switchMode`), и рассинхрон гарантирован, если
делать вручную.

**Заголовки.** Восемь `<h1>` — по одному в тулбаре-логотипе и в каждой модалке
(`:398` newitem, `:445` sprint, `:591` setup, `:659` confirm…). Навигация по заголовкам
выдаёт восемь равнозначных пунктов без иерархии.
→ Оставить **один** `<h1>` (логотип), заголовки модалок сделать `<h2>`.

**Критерий приёмки:** VoiceOver/NVDA объявляет каждый оверлей как диалог с его названием;
выбранный режим просмотра озвучивается.

### X2. Focus management в `LayerManager` · high · 1 день

**Где:** `src/components/layer-manager.js` — файл **целиком** про `z-index`:

```js
// весь LayerManager — это стек + recalculateZ(). Про фокус — ни строки.
```

Grep по `trapFocus|focusTrap|'Tab'` в `src/` → **0**.

Модалки фокусируют свой input (`command-palette.js:28` — `i.focus()`), но:
- **Tab уводит фокус за backdrop** — на элементы под затемнением, которые визуально
  недоступны. Клавиатурный пользователь «проваливается» из диалога и не понимает, где он.
- **После Esc фокус не возвращается** на кнопку, которая открыла диалог — теряется точка
  навигации.

**Сделать — одна правка в `LayerManager`, покрывает все оверлеи:**

```js
open(el) {
  this._prevFocus = document.activeElement;        // запомнить, откуда пришли
  // ...существующая логика z-index...
  const first = el.querySelector(FOCUSABLE);
  if (first) first.focus();
}

close(el) {
  // ...существующая логика...
  if (this._prevFocus) this._prevFocus.focus();    // вернуть фокус
}
```

Плюс **один** `keydown`-обработчик на верхний слой стека: перехватывать `Tab` / `Shift+Tab`
и зацикливать фокус внутри текущего слоя (стандартный focus-trap).

Это ровно тот случай, когда правильная архитектура окупается: `LayerManager` уже
единая точка входа для всех модалок, поэтому фикс не размазывается по десяти файлам.

**Критерий приёмки:** Tab внутри любого оверлея не выходит за его пределы; Esc возвращает
фокус на кнопку-триггер.

### X3. Дедуплицировать разметку card-picker · medium · 4 часа

**Где:** `index.html` — блок пикера (`.ppick` + `.psearch` + `.presults`) продублирован
**12 раз**: `bulk_assigned_pick` (:169), `bulk_iter_pick` (:180), `bulk_parent_pick` (:191),
`s_assigned_pick` (:288), `s_iter_pick` (:299), `s_parent_pick` (:311), `s_deps_*` (:326, :338),
`c_assigned_pick` (:380), `n_*` (:411, :420, :429).

Дрейф **уже начался**: у `bulk_*` пикеров `z-index: 9000`, у `side`/`new` — нет.

Логика уже общая (`card-picker.js`) — а разметка нет. Это значит, что **добавление ARIA
(X1) придётся делать 12 раз**, и это же будет верно для любой следующей правки UX пикера.

**Сделать:** вынести разметку в один `<template>` (или генерировать из JS по id-префиксу)
и клонировать при инициализации. Тогда a11y-атрибуты добавляются **один раз**.

**Порядок важен:** сделать X3 **до** X1, иначе двенадцатикратная работа.

## Часть II. Локализация

### X4. i18n неполон: половина строк захардкожена по-английски · medium · 1 день

**Где:** `setup.js:51,107,225`, `command-palette.js:14-25`, `tutorial-manager.js:131,414`

Словари `en`/`ru`/`de`/`es` есть, движок `window.i18n.t()` работает — но **строки,
генерируемые из JS, мимо него проходят**:

```js
setup.js:51        $('setup-load-hint').innerHTML = SETUP_HINT;   // ← затирает локализованный
                                                                   //   data-i18n-html хардкодом EN
setup.js:225       'Organization is required.'                    // ← ошибки setup по-английски
command-palette.js:14-25   PALETTE_ACTIONS = [...]                // ← вся палитра по-английски
tutorial-manager.js:131    'Welcome to ADO Atlas!'
tutorial-manager.js:414    'Finish' / 'Next'
```

Самое болезненное — **`setup`**. Это **первый экран** и гейт всего приложения: русский,
немецкий или испанский пользователь при первом запуске видит английский текст и английские
ошибки валидации. Первое впечатление о локализованном продукте — что он не локализован.

**Сделать:**
- Правило: **любой** `innerHTML`/`textContent` из JS идёт через `window.i18n.t()`.
  Добавить в гейт (`check-i18n-keys.js` уже существует — расширить его на поиск
  строковых литералов в UI-присваиваниях).
- Ключи для `palette.*` и `tutorial.*` добавить в `src/locales/*.json`.

**Критерий приёмки:** при `lang=ru` экран setup, палитра и туры полностью на русском.

### X5. RTL: честно признать, что его нет · medium · 2 часа (или 1 неделя)

```js
// i18n.js:17
const RTL_LANGS = new Set(/* populate when RTL locales added */);   // ← ПУСТОЙ
// i18n.js:75 — всегда возвращает dir=ltr
```

При этом `ARCHITECTURE.md §10` подаёт RTL-поддержку как **реализованную фичу**.

Вёрстка при этом **физическая**, а не логическая:
```css
views.css:9    left: 1.077rem;          /* не inset-inline-start */
```
`tutorial-manager` позиционирует подсказки по `getBoundingClientRect().left`.

То есть при добавлении `ar`/`he` раскладка **поедет** — RTL не просто не включён, он не
протестирован ни на одной локали.

**Два честных пути:**

- **A (2 часа, рекомендую сейчас).** Убрать заявление о RTL из `ARCHITECTURE.md §10`,
  пометить как «not supported». Арабский/иврит не входят в целевую аудиторию
  ([09-audience.md](09-audience.md) — Microsoft-стек, en/ru/de/es), так что честность
  дешевле реализации.
- **B (неделя, если появится спрос).** Перейти на логические свойства
  (`inset-inline-start` вместо `left`, `margin-inline` вместо `margin-left`), сделать
  dir-aware хелпер, наполнить `RTL_LANGS`.

Чего делать нельзя — **оставить документацию, обещающую RTL**, которого нет.

## Часть III. Дизайн-токены

### X6. 247 захардкоженных hex · medium · фоном · (UNVERIFIED)

**Где:** 13 CSS-файлов

```
base.css:2-7    — ~14 токенов (:root + body.light)   ← вся система токенов
Захардкоженных hex: 247
  ai-search-dialog.css  74
  premium.css           36
  views.css             25
  side-panel.css        23
  base.css              22
Из них: #fff × 51, #333 × 15, золото (#f2a900/#c9a24e), фиолетовый (#a855f7 × 16)
Оверрайдов body.light в компонентных CSS: 18
@media (prefers-color-scheme): 0
```

**Последствие:** каждый новый компонент с baked-цветом нужно **вручную** продублировать
под `body.light`, иначе он не адаптируется к светлой теме (тёмные плашки, нечитаемый текст).
Забыл — получил баг светлой темы. И это происходит при каждом новом компоненте, то есть
техдолг растёт **линейно с продуктом**.

Отдельно: сквозные семейства (PRO-золото, AI-фиолетовый) не вынесены в токены — а значит,
правка бренда = обход десятков файлов.

**Сделать (не спринтом, а по мере касания файлов):**
1. Вынести семейства в токены `base.css`: `--pro-gold`, `--ai-purple`, `--danger`,
   `--success` — каждый с парой значений (dark/light).
2. Заменять hex на `var()` **в тех файлах, которые и так трогаешь**. Не делать отдельный
   мега-рефакторинг CSS — он не окупится.
3. Добавить в гейт grep-проверку: сырой hex вне `base.css` — предупреждение. Это
   останавливает **рост** долга, что важнее, чем разгрести существующий.

**Замечание.** Пункт 3 — самый ценный. Долг в 247 hex терпимо жить, если он перестал расти;
и невыносимо, если каждый новый компонент добавляет ещё десять.
