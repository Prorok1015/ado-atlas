# Спецификация: монетизация, гейтинг, лицензирование

Покрывает: C08, C14, C15, C18, C35, C44, C45.

## Диагноз

Монетизационный контур разомкнут **в трёх независимых местах**. Каждое из них по отдельности
обнуляет выручку, и все три существуют одновременно:

```js
// 1. Кассы нет
entitlement-manager.js:62   async activate(licenseKey) {
                              throw new Error('Premium activation is not available yet');
                            }
premium-paywall.js:139      кнопка покупки — hardcoded disabled

// 2. Гейта нет — gate() определён и НЕ ВЫЗЫВАЕТСЯ НИ РАЗУ (grep: 0 совпадений в src/)
entitlement-manager.js:90   gate(feature) { if (this.isPro()) return true; ... }

// 3. Вместо гейта — безусловный перехват
init.js:281   document.addEventListener('click', (e) => {
                const el = e.target.closest('[data-pro-feature]');
                if (!el) return;
                e.preventDefault();
                if (window.PremiumPaywall) window.PremiumPaywall.open(feature);
                //          ↑ БЕЗ проверки isPro() — Pro-юзер увидит тот же пейволл
              });
```

**Следствие, которое стоит проговорить прямо:** если бы касса заработала прямо сейчас,
человек заплатил бы деньги и **не получил ничего** — при клике на любую Pro-фичу он
упёрся бы в тот же пейволл, что и бесплатный пользователь. Кнопка режима Analytics и вовсе
мертва для всех: `wireControls` делает `if (b.dataset.proFeature) return;`.

И третий слой: **фич, за которые берут деньги, не существует**. `pro-features.js` CATALOG
помечает большинство как `status: 'planned'`; `src/app/analytics.js` (24 строки) — это
**фасад телеметрии**, а не дашборд; `.ultra` CSS в проекте нет вообще.

Итого — витрина с золотыми PRO-бейджами и ценником **$5/мес** поверх функций, которых
нет, при выключенной кассе и неработающем гейте.

## M1. Вызвать `gate()` · high · 1 день · **блокер всего остального**

Метод **уже написан** и уже корректно уважает grace-период и dev-флаг. Его просто забыли
подключить. Это самая дешёвая правка с самым большим рычагом во всём аудите.

**Сделать:** заменить универсальный перехват в `init.js:281` на явные вызовы в точках входа:

```js
// вместо глобального перехвата — в каждой точке входа Pro-фичи:
function openAnalytics() {
  if (!EntitlementManager.gate('analytics')) return;   // free → пейволл, pro → пропуск
  App.settings.switchMode('analytics');
}
```

Три состояния должны заработать по-настоящему:
- **`pro`/`team`** → фича открывается;
- **`free`** → пейволл **с питчем именно этой фичи**;
- **`preview`** → фича работает в демо-режиме (`ProButtonManager.isPreview()`).

Список точек входа = все элементы с `data-pro-feature` (это единственный источник истины
о том, где стоят гейты). Пройти по нему **QA-чеклистом**, ничего не пропустив.

**Обязательно** — покрыть тестом (см. [06-spec-quality-ci.md](06-spec-quality-ci.md#q3)):
`isPro() = true` → `gate()` возвращает `true` и **не** открывает пейволл. Это регрессия,
которая стоит денег, и её нельзя доверять ручному smoke.

**Критерий приёмки:** при `__dev_force_pro = true` все Pro-фичи открываются без пейволла;
кнопка Analytics переключает режим.

Тестовый тумблер **уезжает в собранный ZIP** и остаётся активным навсегда.
`chrome.storage.local.set({__dev_force_pro: true})` из DevTools = все платные фичи, за секунду.

**Сделать:** вырезать флаг в `tools/build.js` для прод-сборки (простая замена
`this._devForcePro` на `false` при билде, либо сборка с `DEBUG=false`).

### Честно о клиентском гейтинге

Даже после M2 **клиентские Pro-фичи принципиально незащитимы**. Это Chrome-расширение:
весь код виден, `chrome.storage` правится из DevTools, `entitlement.status = 'active'`
подделывается за минуту. Аналитика, условное форматирование, темы, экспорт — всё
исполняется в браузере и будет обходиться теми, кто захочет.

**Это нормально, и с этим не надо бороться.** Так живут все расширения. Правильная модель:

- **клиентский `isPro()` — это UX-барьер, а не граница безопасности.** Он честно работает
  для 95% пользователей, которые не полезут в DevTools;
- **настоящий enforcement возможен только для серверных фич** — hosted AI (ключ на сервере),
  cloud sync, тяжёлые отчёты. Именно поэтому [роадмап](10-roadmap.md) ставит Go-бэкенд
  в Фазу 4: это первая фича, которую **нельзя** обойти клиентски;
- **зафиксировать это в `ARCHITECTURE.md` §8/§9** явно, чтобы не строить иллюзий.

Пиратство прайсится в модель, а не побеждается. Для prosumer-аудитории за $6/мес
честные пользователи платят.

## M6. Включить кассу: Lemon Squeezy вместо Go-бэкенда · high · 2 недели

Это уже не столько фикс, сколько **ключевое стратегическое решение роадмапа**, поэтому
обосную его здесь.

`ARCHITECTURE.md` планирует собственный Go-бэкенд для `/api/license/activate` с
device-binding на 3 устройства и 7-дневным grace-периодом. **Модель правильная, а
реализация не нужна.**

**Lemon Squeezy как Merchant of Record закрывает всё это без своего бэкенда:**

| Что нужно | Свой Go-бэкенд | Lemon Squeezy License API |
|---|---|---|
| Приём платежей | писать + PCI | из коробки |
| НДС/налоги десятков юрисдикций | **неподъёмно для соло** | MoR берёт на себя |
| Счета, инвойсы для B2B | писать | из коробки |
| Device-binding (лимит 3) | писать | `instances` — ровно эта модель |
| Dunning (восстановление платежей) | писать | нативно |
| Аптайм, ключи, ops | **на тебе, 24/7** | не твоя проблема |

`activate(licenseKey)` дёргает LS License API (`activate`/`validate`/`deactivate` instances),
где instance = устройство с лимитом 3 — **буквально та модель device-binding, что описана
в `ARCHITECTURE.md §8`**. Фоновый alarm вызывает `validate()`, а уже реализованный
7-дневный grace-период прикрывает офлайн.

**Почему это важнее, чем кажется:** собственный бэкенд — это не только код, это
**операционная нагрузка на одного человека навсегда** (аптайм, ротация ключей, инциденты).
Отложив его до Фазы 4, ты получаешь выручку **на квартал раньше** и без ops.

Go-бэкенд остаётся в плане, но только там, где он **действительно незаменим** —
hosted AI без ключа пользователя (серверный LLM-ключ + rate-limit клиентски не сделать).

**Риск и запасной план:** payout-eligibility Lemon Squeezy для реквизитов автора надо
проверить **до** того, как строить на нём. Держать Paddle/Gumroad как fallback за тонким
слоем абстракции (интерфейс `activate/validate/deactivate` — один и тот же).

## Порядок работ

Строго в этом порядке — каждый пункт разблокирует следующий.

| # | Что | Severity | Оценка | Почему здесь |
|---|---|---|---|---|
| 1 | **M3** — убрать ценник с несуществующего | high | 1 ч | Снимает риск снятия из CWS **сегодня** |
| 2 | **M1** — вызвать `gate()` | high | 1 д | Без него продавать нечего: платный ≠ доступ |
| 3 | **M2** — вырезать `__dev_force_pro` | high | 2 ч | До первой продажи |
| 4 | M4 — один источник тарифа | medium | 3 ч | До того, как фич станет много |
| 5 | **Реализовать 3 Pro-фичи** | — | 3-4 нед | [Фаза 1 роадмапа](10-roadmap.md): условное форматирование, шаблоны, темы |
| 6 | **M6** — Lemon Squeezy checkout | high | 2 нед | Касса — **после** того, как есть что продавать |
| 7 | M5 — консистентный пейволл | medium | 2 ч | Полировка конверсии |

**Ключевая мысль порядка:** сначала «есть что продать» (5), потом «есть чем принять
деньги» (6). Обратный порядок даёт кассу, за которой пустота, — и первые же покупатели
станут первыми же возвратами и однозвёздочными отзывами.
