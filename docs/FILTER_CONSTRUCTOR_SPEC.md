# Конструктор фильтров — спецификация (v0.2 - Упрощенный)

Универсальный визуальный конструктор запросов: пользователь настраивает фильтры по work items через расширенный интерфейс текущих чип-фильтров (pills/bubbles), без знания WIQL. Конструктор расширяет существующий плоский набор фильтров до поддержки любых полей проекта и операторов сравнения, переиспользуя текущие компоненты и классы.

---

## 1. Цель

Расширить текущую чип-панель фильтров до конструктора, который:

- Покрывает **любое** поле ADO (стандартные + кастомные из процесса проекта) через динамическое добавление строк.
- Поддерживает **операторы сравнения** WIQL (=, <>, >, <, >=, <=, Contains, Under) через компактные селекторы.
- Переиспользует **текущие CSS-классы и компоненты** чипов (`.frow`, `.fl`, `.chip`, `.chip.in`, `.chip.out`, `.fclear`).
- Позволяет выбирать значения по клику (neutral -> included/green -> excluded/red -> neutral).
- Является плоским и интуитивно понятным (все строки объединяются через AND, а значения внутри одной строки — через OR/AND), решая 95% реальных задач без перегрузки интерфейса.

## 2. Сравнение подходов

| Параметр | Исходный сложный UI (v0.1) | Упрощенный чиповый UI (v0.2) |
|---|---|---|
| **Структура** | Иерархическое дерево AND/OR групп с вложенностью до 2 уровней | Плоская таблица фильтров (все строки через AND) |
| **Добавление полей** | Создание пустого баббла, выбор поля, выбор оператора, ввод значения | Предустановленные 6 строк + кнопка `[+ Add field…]` для добавления новых строк |
| **Рендеринг чипов** | Каждый чип представляет собой пару "Поле Оператор Значение" | Чипы рендерятся в строках соответствующих полей; клик переключает "включить/исключить" |
| **Режим ввода** | Ввод значения внутри баббла | Поле ввода с селектором оператора отображается в конце строки кастомного поля |
| **Сохранение** | Сложная сериализация дерева | Сохранение плоского `fstate[key] = { value: 'in' | 'out' }` с кодированием оператора в ключе |
| **Переиспользование** | Написание новых CSS/JS компонентов с нуля | Полное переиспользование стилей `.chip.in`, `.chip.out`, `.frow`, `.fl` |

Ключевые принципы:

1. **Filter IR — backend-agnostic.** Использует абстрактные ключи полей (`state`, `assignee`, `priority`), а не вендорные идентификаторы (`System.State`, `status`). Один и тот же IR может быть скомпилирован в WIQL, JQL или любой другой язык запросов.
2. **Все дороги ведут через Filter IR.** Неважно, собрал ли фильтр пользователь руками, сгенерировал ИИ, или загрузил из сохранённого запроса — результат всегда один формат.
3. **Backend Adapter — единственный компонент, знающий целевой язык запросов.** Добавление нового бэкенда (Jira, Linear) = реализация нового адаптера, без изменения IR, UI или AI.

## 4. Filter IR (Intermediate Representation)

### 4.1 Формат

```ts
// Одно условие фильтра
interface FilterCondition {
  id?: string;
  kind: 'condition';
  field: string;              // АБСТРАКТНЫЙ ключ поля: "state", "assignee", "storypoints"
  op: FilterOperator;         // оператор из закрытого списка
  value: FilterValue;         // значение или массив
}

// Группа условий (AND/OR)
interface FilterGroup {
  id?: string;
  kind: 'group';
  logic: 'AND' | 'OR';
  rules: FilterRule[];
}

type FilterRule = FilterCondition | FilterGroup;

interface FilterIR {
  where: FilterGroup;
  orderBy?: OrderByClause[];
}

interface OrderByClause {
  field: string;
  dir: 'asc' | 'desc';
}

type FilterOperator =
  | '='  | '<>' | '>' | '<' | '>=' | '<='  // сравнение
  | 'IN' | 'NOT IN'                         // множество
  | 'CONTAINS' | 'NOT CONTAINS'             // подстрока
  | 'UNDER' | 'NOT UNDER';                  // дерево (Area/Iteration Path)

type FilterValue =
  | string
  | number
  | string[]
  | number[]
  | null;
```

## 5. Реестр полей (FieldRegistry) — backend-agnostic

FieldRegistry — абстрактный каталог полей, доступных для фильтрации. Наполняется BackendAdapter'ом при подключении к конкретному бэкенду.

### 5.1 Абстрактный реестр

```ts
interface FieldSpec {
  key: string;                 // абстрактный ключ: "state", "assignee", "customer_tier"
  displayName: string;         // "State" — для UI
  type: FieldType;             // определяет допустимые операторы и пикер значений
  filterable: boolean;
  sortable: boolean;
  operators: FilterOperator[];
  values?: string[];           // enum-значения
}

type FieldType =
  | 'string'
  | 'integer'
  | 'double'
  | 'dateTime'
  | 'identity'
  | 'treePath'
  | 'html'
  | 'tags';
```

## 6. Backend Adapter

### 6.1 Интерфейс

```ts
interface BackendAdapter {
  readonly id: string;                         // 'ado', 'jira'
  populateFieldRegistry(registry: FieldRegistry): Promise<void>;
  compileQuery(ir: FilterIR, options?: QueryOptions): CompiledQuery;
  executeQuery(compiled: CompiledQuery): Promise<QueryResult>;
  resolveIdentity(name: string): Promise<string | null>;
}
```

## 7. UI: Конструктор на базе Групп-Карточек (Chip-Group Cards)

### 7.1 Быстрые фильтры (Quick Filters)
Панель быстрых фильтров остается на главном экране и сохраняет текущий вид (State, Type, Priority, Assigned, Sprint, Tags).
- Клик по чипу циклически переключает: Включить (зеленый, `.in`) -> Исключить (красный, `.out`) -> Снять.
- Быстрые фильтры компилируются в плоский `FilterIR` (одна корневая группа `AND`).
- Если применен расширенный фильтр, панель быстрых фильтров сворачивается/скрывается, а на ее месте отображается плашка-индикатор:
  `[Активен расширенный фильтр (Редактировать) ✕]`
  Клик на `✕` сбрасывает расширенный фильтр, а клик на плашку открывает диалог конструктора для редактирования.

### 7.2 Расширенный конструктор (Advanced Filter Dialog)
Расширенный конструктор открывается в виде отдельного модального окна (диалога) по кнопке `[Advanced Filters...]`.
Внутри диалога:
1. **Список Групп-Карточек (Filter Group Cards)**:
   - Каждая карточка представляет собой визуальный блок условий, объединяемых по логике **AND**.
   - Внутри карточки пользователь может добавлять строки для любых полей из `FIELD_REGISTRY`.
   - В каждой строке: название поля, кнопка `✕` для удаления строки, активные чипы значений и компактная форма добавления нового чипа (селектор оператора + ввод значения + кнопка `＋`).
2. **Связи между карточками (Группы ИЛИ/И)**:
   - Карточки объединяются по логике **OR** (по умолчанию) или **AND** (через крупный переключатель-разделитель между карточками).
   - Кнопка `[+ Add OR Group]` внизу добавляет новую карточку.
3. **Панель превью результатов (Live Preview)**:
   - Внизу диалога отображается секция превью (список найденных work items с их ID, типом, заголовком и статусом).
   - Превью обновляется на лету (с дебаунсом) при изменении условий в карточках.
   - Пользователь видит количество найденных элементов (например, `Found: 12 items`).
4. **Кнопки управления**:
   - `[Apply Filter]` — закрывает диалог, сбрасывает состояние быстрых фильтров, применяет `FilterIR` к основному экрану и выводит плашку-индикатор активного расширенного фильтра.
   - `[Cancel]` — закрывает диалог без сохранения изменений.

### 7.3 Ввод значений для кастомных полей
- `integer` / `double` -> операторы `=`, `>`, `<`, `>=`, `<=`, `<>`; числовой `<input type="number">`.
- `dateTime` -> операторы `=`, `>`, `<`, `>=`, `<=`, `<>`; выбор даты `<input type="date">`.
- `area` (Area Path) -> операторы `=`, `UNDER`; текстовый автокомплит по путям областей проекта.
- `tags` / `html` / `string` -> операторы `CONTAINS`, `NOT CONTAINS`, `=`; текстовый `<input type="text">`.

## 8. Обратная совместимость и состояние (fstate)

### 8.1 Формат fstate и FilterIR
- Быстрые фильтры сохраняют плоский вид в `fstate`:
  `fstate[fieldKey] = { [value]: 'in' | 'out' }`
- Расширенные фильтры хранятся как структурированный `FilterIR`:
  ```ts
  interface FilterIR {
    where: FilterGroup; // Карточки (Group Cards) как дочерние группы
  }
  ```
- При сохранении состояния в `localStorage`:
  - Если активен быстрый фильтр, сохраняется плоский `ado.filters` (fstate).
  - Если активен расширенный фильтр, сохраняется `ado.filtersAdvanced` (FilterIR), а `ado.filters` сбрасывается.

### 8.2 Синхронизация при загрузке
При старте страницы:
- Если в `localStorage` обнаружен `ado.filtersAdvanced`, мы загружаем расширенный фильтр, скрываем быструю чип-панель и выводим плашку-индикатор.
- Если обнаружен только `ado.filters`, мы загружаем быстрые фильтры и рендерим обычную чип-панель.

## 9. Точка интеграции для AI

AI-поиск генерирует `FilterIR` напрямую.
- Метод `filterConstructor.setIR(ir)` принимает сгенерированный ИИ `FilterIR`.
- Если сгенерированный IR является плоским (только AND), он может быть импортирован как быстрый фильтр.
- Если IR содержит OR-группировки или сложные поля, он импортируется как расширенный фильтр (при этом открывается диалог конструктора для ревью пользователем, быстрые фильтры сбрасываются).

## 10. Специальные фильтры
- `followed`: Рендерится как кнопка "Followed" на панели (в режиме плашки расширенного фильтра кнопка Followed остается доступной рядом).
- `text` (поиск по title): Текстовый ввод на главной панели компилируется в `title CONTAINS 'value'`.
