# Premium / Subscription — Implementation Design

Технический дизайн реализации Freemium-монетизации поверх существующего кода ADO Atlas.
Дополняет `MONETIZATION_AND_ANALYTICS_SPEC.md` (это — требования; здесь — как строим).

---

## 0. Базовые принципы

1. **Клиентский гейтинг не защищается криптографически.** Расширение распаковывается, проверку `isPro()` можно вырезать. Поэтому:
   - **Жёстко на сервере** защищаем только то, что держит реальный секрет или ресурс: **AI-прокси** (ключ к LLM), и опционально Hosted-OAuth token-exchange.
   - **Soft-gate** (honor system) — Analytics-модуль, пресеты фильтров, экспорт. Не тратим силы на обфускацию.
2. **Биллинг-провайдер абстрагирован за нашим бэкендом.** Расширение знает только наш `/api/license/*` и `/api/ai/*`. Смена Lemon Squeezy ↔ Stripe не затрагивает клиент. (Решение на старте: **Lemon Squeezy** — Merchant of Record, лицензионные ключи с лимитом активаций из коробки; миграция на Stripe — когда выручка оправдает налоговую возню.)
3. **Grace period обязателен.** Офлайн / сетевая ошибка валидации НЕ сбрасывает Pro мгновенно — иначе ломаем UX платящим. Pro считается активным, если `status==active` ИЛИ `now - last_validated_at < GRACE_DAYS` (7 дней).
4. **Соблюдаем паттерны проекта** (`.agents/AGENTS.md`): IIFE + экспорт в `global`, оверлеи через `LayerManager`, диалоги через `customConfirm`/`customAlert`, чистый `lib.js`, провайдеры AI через реестр.

## 1. Состояние и хранилище

`chrome.storage.local` (НЕ `sync` — ключ не должен синхронизироваться между устройствами):

```
installation_id   : string  // crypto.randomUUID(), генерится при onInstalled
license_key        : string  // введён пользователем при активации
entitlement        : {
  tier             : 'free' | 'pro' | 'team'
  status           : 'active' | 'past_due' | 'canceled' | 'none'
  expires_at       : number  // epoch ms
  last_validated_at: number  // epoch ms — основа grace period
}
```

## 2. Новые модули

### 2.1 `components/entitlement-manager.js` → `window.EntitlementManager`
Единственный источник истины о тарифе. Синхронные геттеры читают закэшированное состояние (грузится 1 раз при старте + по событиям).

```
init()                 // загрузить entitlement из storage в память
isPro()       : bool   // tier in (pro,team) И (status==active ИЛИ в пределах grace)
getTier()     : string
getStatus()   : string
getExpiry()   : number
activate(key) : Promise // POST /api/license/activate {license_key, installation_id}
refresh()     : Promise // POST /api/license/validate — вызывается из background по alarm
deactivate()           // очистить локальное состояние (logout)
onChange(cb)           // подписка для UI (перерисовать бейджи/гейты)
```

`gate(feature)` helper: если `!isPro()` → открыть paywall-модалку и вернуть `false`; иначе `true`. Точки вызова — см. §4.

### 2.2 `ai/hosted-cloud-provider.js` → `class HostedCloudProvider extends AIProvider`
По образцу `ai/custom-cloud-provider.js`, но:
- не хранит ключ LLM; шлёт `{license_key, installation_id, prompt, context}` на наш бэкенд через `background.js` (`action: "fetchHostedAI"` — новый, по аналогии с `fetchCloudAI`).
- `getAvailability()` → `'available'` только если `EntitlementManager.isPro()`, иначе `'unsupported'`.
- регистрируется в `aiProviderRegistry` всегда, но «отваливается» сам через availability — потребители (`getActive()`) уже умеют это.

### 2.3 Paywall / промо-UI
- Модалка через `LayerManager.open()`, dark-glass промо-экран (Этап 1 спеки).
- Поле ввода `license_key` → `EntitlementManager.activate()`.
- Кнопка «Купить» → внешняя ссылка на checkout Lemon Squeezy.

## 5. Бэкенд (минимальный — на Go)

**Язык бэкенда — Go (Golang).** Один небольшой сервис (stateless, легко хостить — Cloud Run / Fly.io / VPS за reverse-proxy). Нужен в любом случае ради AI-прокси. Эндпоинты:

- `POST /api/license/activate`  `{license_key, installation_id}` → проксирует в Lemon Squeezy `activate` (лимит устройств считает LS). Возвращает `{success, tier, status, expires_at}` или `{error:"DEVICE_LIMIT_EXCEEDED"}`.
- `POST /api/license/validate`  `{license_key, installation_id}` → LS `validate`. → `{valid, status, expires_at}`.
- `POST /api/ai/prompt`         `{license_key, installation_id, prompt, context}` → проверяет лицензию + rate-limit, подставляет серверный ключ LLM, проксирует к OpenAI/Anthropic. Ключ LLM никогда не покидает сервер.

Lemon Squeezy webhooks (`subscription_created/updated/cancelled`) — для своей таблицы статусов/квот, если нужны лимиты AI по тарифу.

**Стек бэкенда (Go):** `net/http` (или chi/echo для роутинга), клиент к Lemon Squeezy License API, клиент к LLM (OpenAI/Anthropic) с серверным ключом из env, простой rate-limiter (in-memory или Redis) на `license_key`. Хранилище статусов — SQLite/Postgres, либо вовсе без БД, если опираться на LS как на источник истины. Секреты (LLM-ключ, LS API key, webhook secret) — только в env бэкенда, никогда в расширении.

## 6. Roadmap (под этот дизайн)

- **Этап 1 (заглушка, без биллинга) — ГОТОВО:** `installation_id` (background `onInstalled`), скелет `EntitlementManager` (free по умолчанию + `__dev_force_pro` для теста, grace period 7 дней), `HostedCloudProvider` (стаб, `BACKEND_LIVE=false`), кнопка Analytics (Pro) с бейджем, paywall-модалка (тема-зависимая, золотой бейдж PRO).
- **Этап 2:** Lemon Squeezy + Go-бэкенд (`activate`/`validate`/`ai`), включение `HostedCloudProvider` (`BACKEND_LIVE=true` + своя карточка в AI-диалоге), фоновая валидация по alarm + grace period, домен бэкенда в manifest (`host_permissions` + CSP `connect-src`).
- **Этап 3:** Analytics-модуль (Revisions API → Cycle Time / Flow), пресеты фильтров, экспорт.
- **Этап 4:** sandbox-тесты платежей, модерация в Chrome Web Store.

## 9. Правовая и платёжная сторона (приём денег + Chrome Web Store)

> ⚠️ **Дисклеймер:** это не юридическая/налоговая консультация, а ориентир. Конкретика зависит от **страны вашего налогового резидентства** (она здесь неизвестна) и меняется со временем. Перед запуском платежей сверьтесь с местным бухгалтером/юристом.

### А. Нужно ли открывать компанию, чтобы принимать платежи?

Коротко: **чтобы начать — компания не обязательна.** Длинно:

1. **Merchant of Record (Lemon Squeezy / Paddle) радикально снижает правовой порог.** Юридически продавцом конечному клиенту выступает MoR: он выставляет счёт, собирает и платит налог с продаж (VAT/GST/Sales Tax) по миру. Вы получаете **выплаты (payout)** от MoR — это ваш доход, а не «розничная торговля софтом по всему миру». Принимать выплаты можно как **физлицо / самозанятый / ИП** (зависит от страны и порога дохода).
2. **Ваш доход всё равно облагается налогом по месту резидентства.** MoR закрывает *налог с продаж покупателю*, но **не** ваш подоходный/налог на прибыль. Регулярный доход почти везде требует легализации статуса: самозанятый / ИП / ФОП / sole trader / Einzelunternehmer и т.п. Где-то есть необлагаемый порог (hobby income), выше — обязательная регистрация.
3. **Способ получить деньги.** Нужен счёт/кошелёк, который поддерживает ваш MoR (обычно банковский перевод, PayPal, Wise). Проверьте, что MoR делает выплаты в вашу страну (санкционные/ограниченные юрисдикции бывают исключены).
4. **Когда открывать ООО/LLC/LTD:** ради ограничения ответственности, оптимизации налогов, найма, B2B-договоров и инвойсов на Team/Enterprise-тариф. Это разумно **позже**, когда выручка оправдает бухгалтерию и сборы. На старте — оверкилл.
5. **Если откажетесь от MoR в пользу прямого Stripe:** вы становитесь merchant of record сами → регистрация плательщиком VAT/GST по порогам в каждой юрисдикции, подача деклараций, скорее всего нужна зарегистрированная компания. Порог входа существенно выше (см. §0/§выбор биллинга).

**Рекомендация для соло-старта:** самозанятый/ИП (если страна требует) + **Lemon Squeezy/Paddle**. Компанию — по мере роста.

### Б. Что включить/настроить в Google-аккаунте паблишера, чтобы не словить блок

1. **Developer-аккаунт:** одноразовый регистрационный взнос (~$5), подтверждённый email, **включённый 2FA** на Google-аккаунте (риск угона = риск потери стора).
2. **Trader status (статус «трейдер») — обязателен для монетизации.** По EU DSA в Chrome Web Store нужно указать, что вы «trader», и опубликовать **контакты**: имя/название, физический адрес, email, телефон. Без корректного trader-раздела платные/монетизируемые расширения **снимают с публикации**. Контакты должны быть реальными и актуальными.
3. **Платежи — вне магазина.** Встроенных платежей CWS больше нет: оплата идёт через ваш процессор (LS/Paddle/Stripe), лицензирование — через наш бэкенд (ровно как в этом дизайне). Стор требует не платёжную интеграцию, а **честные дисклоузеры**.
4. **Вкладка Privacy practices (критично для модерации):**
   - Ссылка на **Privacy Policy** (у проекта есть `PRIVACY.md` / `privacy.html`) — обновить под новые данные: email при покупке, `license_key`, `installation_id`, проксирование AI-запросов на наш сервер.
   - **Обоснование каждого permission** и `host_permissions` (включая новый домен бэкенда). Не запрашивать лишнего.
   - Сертификация **Limited Use**: данные используются только для заявленной функции, не продаются, не идут на рекламу/скоринг.
5. **MV3 / Remote code:** запрещено исполнять удалённый код. Наш AI-прокси гоняет **данные** по сети (это разрешено), но **нельзя** подгружать и выполнять внешний JS. CSP уже строгий — не ослаблять; новые домены только в `connect-src`/`host_permissions`.
6. **Single purpose:** расширение должно иметь одну узкую цель. Аналитика/AI допустимы как продолжение основной функции (визуализация ADO), не как сборная солянка.
7. **Честность оффера (иначе «deceptive behavior»):** в описании стора и в UI явно указать: это платная подписка, что **бесплатно**, что в **Pro**, цена, отмена/возврат, контакт поддержки. Базовый функционал не запирать (модель freemium — соблюдается).
8. **GDPR/приватность по существу:** раз собираем email + `installation_id` и проксируем запросы — privacy policy должна описывать обработку, хранение и удаление; для Team/Enterprise (B2B) может потребоваться DPA. Бэкенд: секреты только в env, логи без лишних персональных данных.

### В. Что включить на стороне расширения (manifest/UI), чтобы пройти ревью

- `manifest.json`: минимум permissions; новый домен бэкенда — в `host_permissions` **и** CSP `connect-src`; ссылка на privacy policy; точное `description`.
- Paywall и тексты — не вводят в заблуждение; ключ активируется честно; есть контакт поддержки.
- Бесплатный режим работает без оплаты (freemium-разделение из §2 спеки монетизации).
- Обновить `PRIVACY.md`/`privacy.html` и `STORE_LISTING.md` под платную модель и новые данные **до** первой платной публикации.

### Г. Чек-лист перед включением платежей
1. Определить налоговое резидентство и при необходимости зарегистрировать самозанятость/ИП.
2. Завести аккаунт MoR (Lemon Squeezy/Paddle), настроить продукт/подписку и payout-метод.
3. Включить 2FA, заполнить trader-контакты и Privacy practices в CWS.
4. Обновить privacy policy + store listing под подписку и новые данные.
5. Прогнать сценарий «покупка → выдача ключа → активация → валидация» в sandbox.

### Д. Заметка по юрисдикции основателя (рабочая гипотеза + что проверить)

> ⚠️ Не юр./налоговая консультация. Профиль трансграничный (гражданство РФ, основной счёт в Армении, фактическое проживание в Таиланде по DTV-визе) — обязательно проверить с профильным трансграничным налоговым консультантом **до** регистрации бизнеса и включения платежей.

**Не путать три независимые вещи:**
1. **Налоговое резидентство** — определяется фактом физического присутствия (обычно 183+ дня) и центром жизненных интересов, а **не** местом регистрации бизнеса.
2. **Место регистрации бизнеса** (рабочая гипотеза — **ИП в Грузии** + Small Business Status, 1% с оборота до ~500k GEL/год, далее 3%).
3. **Кто реально проведёт выплаты** — платёжный процессор / MoR с KYC на бенефициара. **Вероятно, главный ограничитель.**

**Открытые вопросы для консультанта/процессора (в порядке приоритета):**
1. **Платёжный канал (проверять ПЕРВЫМ):** поддерживает ли выбранный MoR (Lemon Squeezy/Paddle) Грузию как страну бизнеса; проведёт ли payout на армянский/грузинский счёт; пройдёт ли онбординг владелец-**гражданин РФ** (санкционный скрининг бенефициара). Stripe в Грузии/Армении напрямую недоступен.
2. **Таиланд + 183 дня:** присутствие в Таиланде 183+ дня в году → возможное налоговое резидентство Таиланда (с 2024 облагается доход, **ввезённый** в страну). Может конфликтовать с грузинским режимом.
3. **Substance грузинского ИП:** грузинский 1% обычно предполагает грузинское налоговое резидентство / реальное «существо»; при фактическом проживании в другой стране режим может быть оспорим.
4. **Обязанности гражданина РФ:** уведомление о зарубежных счетах/КИК, валютные правила (даже при статусе нерезидента РФ по 183-дневному критерию).

**Принцип:** сначала подтвердить, что деньги физически дойдут (п.1), и только потом фиксировать юрисдикцию бизнеса.
