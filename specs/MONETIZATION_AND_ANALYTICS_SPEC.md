# Техническое задание (ТЗ): Монетизация и Аналитика для ADO Atlas

Этот документ является полным техническим заданием на проектирование, разработку и интеграцию системы монетизации (Freemium, подписка $5/мес) и аналитического модуля для Chrome-расширения ADO Atlas.

---

## 1. Общие сведения и бизнес-цели
* **Цель проекта:** Внедрение модели Freemium для монетизации ADO Atlas через подписку за $5/мес (или $49/год) за счет предоставления продвинутых QoL-функций и инструментов процессной аналитики (Flow Metrics, Cycle Time).
* **Целевая аудитория Premium:** Team Leads, Scrum Masters, Agile Coaches, Project/Engineering Managers.
* **Целевая аудитория Free:** Разработчики и менеджеры, использующие базовые инструменты визуализации.
* **Принцип разделения:** Базовый функционал (Tree, Graph, Board, Timeline) остается полностью бесплатным и неограниченным. Дополнительные аналитические возможности и QoL-улучшения для команд продаются по подписке.

## 3. Требования к аутентификации и авторизации в Azure DevOps

В связи с отключением Global PATs (1 декабря 2026 года) и устареванием Legacy OAuth, ADO Atlas поддерживает следующие методы:

1. **Organization-Scoped PAT (Free):**
   * Пользователь генерирует токен, ограниченный конкретной организацией, и вводит его вручную.
2. **BYO-Client-ID Entra OAuth (Free):**
   * Разработчик регистрирует свое приложение в своем Microsoft Entra ID по нашей инструкции и указывает Client ID в настройках. Без затрат с нашей стороны.
3. **Hosted/Managed OAuth (Pro):**
   * Расширение использует единый Client ID центрального приложения ADO Atlas, зарегистрированного нами в Entra ID. Вход происходит в один клик.
4. **Session Cookie Auth (Резервный метод):**
   * Использование сессионных кук домена `dev.azure.com` (при наличии согласия пользователя на пермишен `cookies`). Позволяет использовать расширение без ввода паролей и токенов, если пользователь залогинен в Azure DevOps в Chrome.

## 4. Спецификация серверной части и Защита от утечки ключей

Чтобы предотвратить использование одного лицензионного ключа на множестве устройств разными людьми, внедряется механизм **Device Binding (привязка к устройству)**. 

### А. Принцип защиты от утечки ключей
1. При первом запуске расширение генерирует случайный уникальный идентификатор инсталляции — `installation_id` (UUID v4) и сохраняет его в `chrome.storage.local`.
2. При активации лицензии расширение отправляет на сервер и `license_key`, и `installation_id`.
3. Сервер привязывает данный `installation_id` к лицензии.
4. **Лимит активаций:** Для одной лицензии (тарифа Pro) разрешено максимум **3 активных устройства** (например, рабочий ПК, домашний ПК, ноутбук). Если лимит превышен, сервер отклоняет активацию с ошибкой: *"Превышен лимит устройств"*.
5. При ежесуточной валидации расширение отправляет `license_key` + `installation_id`. Сервер проверяет не только дату окончания подписки, но и то, привязано ли текущее устройство к этому ключу.

### Б. Вариант реализации 1: Google Firebase (Firestore + Cloud Functions)

Использование Firebase позволяет быстро развернуть бэкенд без настройки серверов баз данных и Docker.

#### 1. Структура базы данных Firestore
* **Коллекция `licenses`:**
  * Документ: `{license_id}` (может быть равен `license_key` или случайно генерироваться)
  * Поля:
    * `license_key`: `string`
    * `email`: `string`
    * `stripe_customer_id`: `string`
    * `stripe_subscription_id`: `string`
    * `status`: `string` ('active', 'canceled', 'past_due')
    * `expires_at`: `timestamp`
    * `active_activations`: `array of strings` (список привязанных `installation_id`)
    * `created_at`: `timestamp`

#### 2. Бессерверные функции Firebase Cloud Functions
* `stripeWebhook`: Обработчик вебхуков от Stripe.
  * При покупке: генерирует `license_key`, создает документ в Firestore, отправляет ключ на email покупателя.
  * При продлении/отмене: обновляет `status` и `expires_at` в Firestore.
* `activateLicense` (HTTPS-эндпоинт):
  * Принимает `license_key` и `installation_id`.
  * Если `installation_id` уже есть в массиве `active_activations` для этого ключа — возвращает успех.
  * Если `installation_id` новый: проверяет, что `active_activations.length < 3`. Если да — добавляет ID в массив и возвращает успех. Если нет — возвращает ошибку лимита.
* `validateLicense` (HTTPS-эндпоинт):
  * Принимает `license_key` и `installation_id`.
  * Проверяет, что лицензия `active` (или `expires_at > now`) и данный `installation_id` присутствует в массиве `active_activations`.

### В. Вариант реализации 2: Свой сервер (Node.js / Cloudflare Workers / Firebase)

Поскольку расширению требуется проксировать AI-запросы для Premium пользователей без раскрытия API-ключей LLM в клиентском коде, **создание собственного бэкенда является стратегически обязательным**.
* Бэкенд выполняет роль единого хаба для управления лицензиями (учет лимитов на устройства) и маршрутизации AI-запросов к внешним провайдерам (OpenAI/Anthropic).
* Платежи обрабатываются через интеграцию с Lemon Squeezy или Stripe по модели Webhooks. При успешной оплате биллинг-провайдер дергает вебхук на нашем сервере, сервер генерирует ключ и отправляет его пользователю.
* В самом расширении ключи от LLM не хранятся, а API-вызовы ИИ происходят только при наличии валидного `license_key` и `installation_id`.

---

### Г. Спецификация API (Собственный сервер)

#### 1. `POST /api/license/activate`
* **Запрос:**
  ```json
  {
    "license_key": "XXXX-XXXX-XXXX-XXXX",
    "installation_id": "uuid-v4-generated-by-extension"
  }
  ```
* **Ответ (Успех):**
  ```json
  {
    "success": true,
    "expires_at": "2026-07-13T22:00:00Z",
    "status": "active"
  }
  ```
* **Ответ (Ошибка лимита):**
  ```json
  {
    "success": false,
    "error": "DEVICE_LIMIT_EXCEEDED",
    "message": "Превышен лимит устройств для данного ключа. Пожалуйста, деактивируйте старые устройства."
  }
  ```

#### 2. `POST /api/license/validate`
* **Запрос:**
  ```json
  {
    "license_key": "XXXX-XXXX-XXXX-XXXX",
    "installation_id": "uuid-v4-generated-by-extension"
  }
  ```
* **Ответ:**
  ```json
  {
    "valid": true,
    "status": "active",
    "expires_at": "2026-07-13T22:00:00Z"
  }
  ```

#### 3. `POST /api/ai/prompt` (AI Proxy)
* **Запрос:**
  ```json
  {
    "license_key": "XXXX-XXXX-XXXX-XXXX",
    "installation_id": "uuid-v4-generated-by-extension",
    "prompt": "Пользовательский промпт",
    "context": "Контекст из Azure DevOps (опционально)"
  }
  ```
* **Ответ (Успех):**
  ```json
  {
    "success": true,
    "response": "Ответ от LLM"
  }
  ```
* Сервер сам подставляет свой серверный API-ключ от провайдера (OpenAI/Anthropic) и перенаправляет запрос, проверяя лимиты Rate Limit (квоты) для конкретного `license_key`.

## 5. Поэтапный план реализации (Roadmap)

### Этап 1: Подготовка расширения и Заглушка (1-2 дня)
* Добавить в верхний тулбар кнопку-переключатель **«Analytics 👑»**.
* При клике на неё рендерить красивый промо-экран (Dark Mode, стеклянный эффект, скриншоты будущих графиков) с кнопкой «Активировать Premium» и полем ввода лицензионного ключа.
* Реализовать генерацию `installation_id` (UUID) при первом запуске расширения и сохранение его в `chrome.storage.local`.

### Этап 2: Интеграция биллинга и валидатора (2-3 дня)
* **При выборе Firebase:**
  * Настроить проект в консоли Firebase.
  * Развернуть Cloud Function с интеграцией Stripe SDK и логикой привязки `installation_id` в Firestore.
* **При выборе Lemon Squeezy:**
  * Зарегистрировать аккаунт, настроить подписку и включить генерацию лицензионных ключей (макс. 3 активации).
* **В расширении:**
  * Реализовать отправку запросов на активацию и фоновую ежедневную валидацию с передачей `installation_id`.
  * При успешной валидации устанавливать состояние премиума, при ошибке — сбрасывать в `false`.

### Этап 3: Разработка премиум-фич (3-5 дней)
* **Аналитический модуль:**
  * Реализовать парсинг истории изменений задач из API Azure DevOps (Revisions API).

  * Интегрировать графики (с использованием легкой библиотеки вроде Chart.js или SVG-отрисовки напрямую).
* **QoL улучшения:**
  * Добавить возможность сохранения фильтров в `chrome.storage.local`.
  * Реализовать экспорт Gantt-таймлайна в PNG/SVG.

### Этап 4: Тестирование и запуск
* Тестирование платежей в режиме Stripe Sandbox / Lemon Squeezy Sandbox.
* Модерация обновления в Chrome Web Store.

## 6. Стратегия масштабирования бэкендов (Jira, Trello, Asana и др.)

При добавлении поддержки других платформ управления задачами (Jira, Trello, Asana и др.) следует придерживаться модели **"Free Backend Core + Paid Analytics/QoL"**:

### А. Почему базовые бэкенды должны быть бесплатными:
* **Интеграции как источник трафика (SEO):** Поддержка популярных платформ привлекает огромный поток бесплатных установок из Chrome Web Store от людей, ищущих визуальные инструменты (например, "Jira Gantt Chart").
* **Унификация воронки продаж:** Вместо того чтобы продавать саму Jira, мы предлагаем ту же ценность — расширенную аналитику и QoL-фичи поверх любой из платформ.

### Б. Платные B2B / Enterprise функции интеграций:
Некоторые сложные возможности интеграции могут быть перенесены в Premium:
1. **Multi-Source доски:** Отображение задач из разных платформ (например, Jira + Trello) на одном Gantt-таймлайне.
2. **Кросс-платформенная синхронизация:** Автоматический перенос изменений между разными сервисами.
3. **Jira Enterprise:** Поддержка локальных/закрытых серверов (Jira Data Center / Jira Server за VPN) доступна только в Premium, в то время как стандартная Jira Cloud бесплатна.
